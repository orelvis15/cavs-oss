The problem

Small content changes
can become huge downloads.

Game releases are usually distributed as large binary artifacts: PCKs, bundles, packs, archives or asset containers. When a few assets change, players often still download a large portion of the release again.

Traditional full-file downloads are simple, but waste bandwidth. Pairwise binary deltas can be tiny, but they create operational complexity when you need to support many version jumps, resumable downloads, CDN delivery and persistent cache reuse.

  • Players re-download bytes they already have.
  • Launchers need to support many possible version jumps.
  • CDN / object storage gets expensive with duplicated builds.
  • Pairwise deltas scale poorly across many versions.
  • Corrupt or interrupted downloads need safe recovery.
What CAVS does

CAVS turns releases into
verified, reusable chunks.

CAVS packages a release into content-addressed chunks, signs and verifies the manifest, serves only the chunks a client is missing, and reconstructs the original file byte-for-byte on disk.

For cold installs, CAVS can use a full bootstrap artifact when it is cheaper. The client then seeds its chunk cache from that install, so the next update is incremental — without any extra download.

Hybrid reconstruction — v0.6.0

The version already on disk
is now a source of bytes.

A player with the old build installed but an empty cache used to pay the full download. In v0.6.0 CAVS copies verified ranges straight from the old file and fetches only what changed — every range BLAKE3-checked before it is written, the final output still gated by SHA-256.

// previous artifact

Reuse the old install

Point the client at --previous-artifact. Shared ranges are copied from disk, not the network — no chunk cache required.

// signatures

Compact .cavssig

A deterministic old-version signature, ~0.07% of the source (88 KiB for a 128 MiB build), lets a new version be diffed without the old bytes.

// no-op

Skip unchanged work

An output that already matches costs 0 payload bytes. In directory mode, unchanged files — including player mods — are never rewritten.

// coalescing

Fewer, larger reads

Adjacent copy ranges merge: on the shifted build, 1,082 chunk ops → 18 contiguous reads, read amplification 1.0.

No regression: with a warm cache the wire cost is byte-for-byte identical to v0.5, and a corrupt old install demotes to the network per range instead of failing — the output is always byte-identical.
Offline toolkit — v0.7.0

Sign, preview, diff, apply
and verify — no server.

v0.7.0 turns CAVS into a complete local toolkit. Describe a release with a compact .cavssig, see exactly what the next build changes, produce a deterministic .cavsplan patch, and apply it with the same verified reconstruction the online client uses — staged, journaled, byte-identical or it fails.

// preview

See the change first

cavs preview classifies every entry NEW / MODIFIED / DELETED / SAME, estimates the update per route, and warns when a compressed file will patch badly.

// diff-plan

Portable .cavsplan

A deterministic, BLAKE3-sealed patch: copy-ranges from the old build plus zstd-19 fresh data. Diff against just the old signature — no old bytes needed.

// apply

Safe staged apply

cavs apply --inplace stages, verifies every file, journals, then commits. Interrupt it and re-run; mods and unchanged files are left untouched.

// verify-install

Verify any install

cavs verify-install checks a folder against a signature, reporting MODIFIED / MISSING / EXTRA — with --allow-extra-files for saves and mods.

How it works

How CAVS delivers less.

01

Package

cavs pack splits the release into content-defined chunks, hashes each with BLAKE3, stores a compact manifest, and optionally emits a bootstrap artifact.

02

Plan

At session open, the server compares the client's cache state with the manifest. Cold clients may get the bootstrap route; warm clients get only missing chunks.

03

Transfer

Missing chunks are served in binary CVSP batches. Packfile-backed stores coalesce nearby chunk reads into a few range reads.

04

Verify

The client verifies chunk hashes, streams reconstruction to disk, checks the final SHA-256, then atomically promotes the result.

05

Reuse

The reconstructed file seeds the persistent cache — and, in v0.6.0, the version already installed on disk is itself a source of verified ranges, so an update reuses local bytes before it fetches any.

Quickstart

Try it locally.

Run CAVS against two versions of your build and compare full download vs update payload.

1 — Install

$ cargo install cavs-cli
$ cargo install cavs-server
$ cargo install cavs-client

2 — Pack two versions

$ cavs pack --raw game_v1.pck \
    --profile auto --bootstrap \
    -o game_v1.cavs

$ cavs pack --raw game_v2.pck \
    --profile auto --prev game_v1.cavs \
    -o game_v2.cavs

3 — Serve

$ cavs-server game_v1.cavs game_v2.cavs \
    --listen 127.0.0.1:8990

4 — Fetch & check stats

$ cavs-client fetch http://127.0.0.1:8990 game_v2 \
    -o out/game_v2.pck --cache ./cache \
    --stats-json stats.json

# { "delivery_mode": "chunks",
#   "payload_bytes": 1719664,
#   "reconstructed_sha256_ok": true }

5 — v0.6.0: update reusing the old install (empty cache)

$ cavs-client fetch http://127.0.0.1:8990 game_v2 \
    -o out/game_v2.pck --cache ./cache \
    --previous-artifact ./game_v1.pck --stats-json stats.json

# { "sources": { "previous_artifact_bytes": 120974668,
#                "network_bytes": 6544479 },   ← −90% vs a cold v0.5 fetch
#   "reconstructed_sha256_ok": true }

6 — v0.7.0: offline patch — no server (sign · preview · diff · apply · verify)

$ cavs signature export ./Build_v1 --raw -o build_v1.cavssig
$ cavs preview ./Build_v2 --against build_v1.cavssig --changes-only
$ cavs diff-plan ./Build_v1 ./Build_v2 -o update.cavsplan
$ cavs apply --old ./InstalledGame --plan update.cavsplan --inplace --verify

Honest limitations

Where CAVS is not
the right tool.

CAVS is not a video codec and does not pretend to replace one. It's most valuable when redundancy crosses versions, releases, bundles or repeated assets.

  • Single video, first play: ~0% savings, small packaging overhead.
  • ABR ladders: different bitrates usually share no bytes.
  • Already-compressed JPEG / MP4 / ZIP files: near-zero savings.
  • Pairwise delta tools can be smaller for one exact version pair.
CAVS is strongest for game updates, engine packs, launchers, DLC, binary assets and version streams where cache reuse matters.
